# AI Glasses V2 — RK3576 Chip-Down (event-driven, always-on + compute-island)

This folder holds the **V2 design**, a clean break from the V1 CM4-carrier board.

## The one-sentence definition

> V2 is **not** "glasses that run RK3576 all day."
> V2 is **glasses whose low-power always-on (AON) system stays awake all day for sensing + connectivity, and wakes RK3576 for short bursts** when vision / AI / encode / high-rate transmit is actually needed.

This inverts the V1 order. V1 built the RK3576/CM4 host first and treated power as an afterthought (`1S → 5V boost → everything`). V2 **freezes the battery volume and the power state machine first, then draws the RK3576 chip-down** as a demand-started compute accelerator.

## Why the change (energy accounting)

V1 measured/estimated draw: idle **651 mW**, mixed **~3031 mW**, AI+record peak **~4.8 W**. At those numbers a temple-sized battery cannot deliver all-day standby *and* multi-hour mixed use. V2 targets:

| State | V1 now | V2 target |
|---|--:|--:|
| Standby | 651 mW | **20–50 mW** |
| Mixed use | 3031 mW | **300–500 mW** |
| Continuous 1080p record | ~3 W | **1.0–1.5 W** |
| AI + record peak | 4.8 W | **1.5–2.5 W** |

## Status / what's frozen vs open

- **Frozen intent:** event-driven architecture, AON-controls-SoC-power, distributed 4-board split, per-island current sense, hardware default-OFF enables.
- **Now present:** generated pre-layout KiCad system schematic: **96 components**, **127 declared nets**, ERC clean. It is a functional-block review artifact, not a PCB layout release.
- **Open / HOLD:** NDP120 full datasheet + kit + power, FCU760K Hardware Design + BSP, LP451165 + 1S2P validation, custom IMX415 module rail/FPC data, real battery-bay fit.
- **Gate order:** Phase 0 (freeze battery + state machine) → Phase 1 (AON board) → Phase 1.5 (mechanical/electrical floorplan) → Phase 2 (RK3576 min core). PCB layout is still blocked by [docs/12_layout_entry_gate_status.md](docs/12_layout_entry_gate_status.md) and [docs/13_mechanical_electrical_floorplan.md](docs/13_mechanical_electrical_floorplan.md).

## How to regenerate the V2 schematic/docs

```bash
cd "AI Glasses"
.venv/bin/python v2_chipdown/scripts/chipdown_bom.py
.venv/bin/python v2_chipdown/scripts/generate_chipdown_sch.py
.venv/bin/python v2_chipdown/scripts/generate_gate_docs.py
.venv/bin/python v2_chipdown/scripts/generate_bom_status.py

KCLI=/Applications/KiCad/KiCad.app/Contents/MacOS/kicad-cli
"$KCLI" sch erc --severity-all --format report \
  -o v2_chipdown/reports/erc.report.txt \
  v2_chipdown/hardware/ai_glasses_v2_chipdown.kicad_sch
"$KCLI" sch export netlist --output v2_chipdown/reports/ai_glasses_v2.net \
  v2_chipdown/hardware/ai_glasses_v2_chipdown.kicad_sch
"$KCLI" sch export bom --output v2_chipdown/reports/ai_glasses_v2_bom.csv \
  --fields Reference,Value,BOM_ID,Board,Assembly,Priority,Package,Gate \
  v2_chipdown/hardware/ai_glasses_v2_chipdown.kicad_sch
"$KCLI" sch export pdf --output v2_chipdown/reports/output/ai_glasses_v2_chipdown_schematic.pdf \
  v2_chipdown/hardware/ai_glasses_v2_chipdown.kicad_sch
"$KCLI" sch export svg --output v2_chipdown/reports/output \
  v2_chipdown/hardware/ai_glasses_v2_chipdown.kicad_sch
```

## Relationship to V1

V1 (`hardware/ai_glasses_carrier.*`, `scripts/carrier_bom.py`, `config/*.yaml`) stays as the **bench/EVT development vehicle** — it is still the fastest way to bring up RK3576 Linux and get *real* per-rail current numbers. Those measured numbers feed back into `config/power_budget_v2.yaml` and re-size the battery. V2 does **not** delete V1; it supersedes it as the product architecture.

> All numbers in `config/*.yaml` are **engineering estimates** from datasheet typicals until Phase 2 bench EVT replaces them with per-island shunt measurements.
